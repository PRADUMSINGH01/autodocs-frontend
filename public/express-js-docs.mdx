This documentation provides an in-depth, professional overview of the `express` repository, a fast, unopinionated, and minimalist web framework for Node.js. It covers architectural design, API specifications, usage patterns, security considerations, performance optimizations, and guidelines for contributing to the project.

---

## 1. Executive Summary

Express.js is the de facto standard web framework for Node.js, offering a robust set of features for building single-page applications, websites, hybrid applications, and public HTTP APIs. Currently at version `5.2.1`, Express maintains its philosophy of providing small, powerful tooling, allowing developers the flexibility to craft their ideal framework without imposing specific ORMs or template engines. Its core strengths lie in its high performance, extensive test coverage, and a lean, extensible architecture.

This documentation serves as the ultimate professional guide for developers looking to leverage Express.js for their projects, understand its internal workings, contribute to its development, or evaluate its suitability for demanding applications requiring strong security and performance characteristics.

## 2. Architecture Deep-Dive

Express.js is built around a modular and extensible architecture, centered on a core `Application` instance that dispatches requests through a configurable chain of middleware and route handlers. The framework leverages Node.js's native HTTP server capabilities, enhancing them with powerful routing, helper methods, and extensibility points.

The core components and their interactions are as follows:

1.  **Application (`app`)**:
    *   The central orchestrator, typically created by `express()`.
    *   Manages server-wide configurations via `app.set()` (e.g., `view engine`, `trust proxy`, `etag`).
    *   Registers middleware and routes using `app.use()` and HTTP method-specific functions like `app.get()`, `app.post()`, `app.all()`.
    *   Handles incoming requests through its `app.handle()` method, which acts as the entry point for the request-response cycle.
    *   Provides a view rendering mechanism with `app.render()`.
    *   Can be mounted as a sub-application within another Express application, inheriting settings and prototypes.

2.  **Router (`express.Router`)**:
    *   A mini-application itself, capable of handling middleware and routes.
    *   Allows for modular, mountable route handlers, facilitating code organization (e.g., `app.use('/api', apiRouter)`).
    *   Each `app.VERB()` call creates an internal `Route` object managed by the `Router`.
    *   Supports parameter pre-processing via `app.param()` callbacks, which are executed once for a given parameter value in a request path.
    *   Handles various path patterns, including dynamic parameters (`:name`), optional segments (`{/:op}`), and wildcards (`*name`, `+name`).

3.  **Route (`express.Route`)**:
    *   Represents a single path definition for a specific HTTP method (e.g., `GET /users`).
    *   Can have multiple handlers associated with it, forming a mini-middleware stack that executes sequentially for that specific route.
    *   Handles errors using arity-4 functions (`(err, req, res, next)`).

4.  **Request (`req`)**:
    *   An extension of Node.js's native `http.IncomingMessage`.
    *   Decorated with numerous Express-specific properties and methods to simplify common tasks:
        *   **Header Access**: `req.get()`, `req.header()`.
        *   **Content Negotiation**: `req.accepts()`, `req.acceptsCharsets()`, `req.acceptsEncodings()`, `req.acceptsLanguages()`, `req.is()`.
        *   **Request Details**: `req.protocol`, `req.secure`, `req.ip`, `req.ips`, `req.hostname`, `req.path`, `req.subdomains`.
        *   **Query String**: `req.query` (parsed based on `query parser` setting).
        *   **Caching**: `req.fresh`, `req.stale`.
        *   **Parameters**: `req.params` (extracted from the route path).

5.  **Response (`res`)**:
    *   An extension of Node.js's native `http.ServerResponse`.
    *   Provides a rich set of methods for sending various response types and manipulating response headers:
        *   **Status & Headers**: `res.status()`, `res.sendStatus()`, `res.set()`, `res.append()`, `res.get()`, `res.vary()`, `res.links()`.
        *   **Content Sending**: `res.send()` (intelligent content-type based on input), `res.json()`, `res.jsonp()`, `res.sendFile()`, `res.download()`.
        *   **Cookies**: `res.cookie()`, `res.clearCookie()`.
        *   **Redirection**: `res.location()`, `res.redirect()`.
        *   **Content Negotiation**: `res.format()`.
        *   **View Rendering**: `res.render()`.

6.  **Middleware**:
    *   Functions that have access to the request (`req`), response (`res`), and the next middleware function in the application's request-response cycle (`next`).
    *   Can perform tasks like: executing code, making changes to the request and response objects, ending the request-response cycle, or calling `next()` to pass control to the next middleware function.
    *   Error-handling middleware are distinguished by having an arity of 4: `(err, req, res, next)`. These are invoked when `next(err)` is called.
    *   Express includes several built-in middleware for common tasks, such as `express.json()`, `express.urlencoded()`, `express.raw()`, `express.text()`, and `express.static()`.

**Interaction Flow**:

An incoming HTTP request is received by the Node.js `http.Server`, which then delegates to `app.handle()`. The `app.handle()` method first applies any global settings (like `X-Powered-By`) and then passes the request and response objects to the `Router`.

The `Router` iterates through its stack of middleware and routes. For each entry, it checks if the path matches the incoming request URL. When a match is found:
1.  Any `app.param()` callbacks for parameters in the path are executed.
2.  The middleware/handler function is invoked.
3.  If `next()` is called, control passes to the next matching entry in the stack.
4.  If a response method (e.g., `res.send()`, `res.end()`) is called, the response is finalized, and the cycle ends.
5.  If `next(err)` is called, the router bypasses regular middleware and passes control to the next error-handling middleware in the stack.

The design emphasizes a flexible, unopinionated approach, allowing developers to integrate various third-party modules or create custom middleware to extend functionality as needed.

## 3. API Reference

This section details the primary interfaces, methods, and types available in Express.js.

### `express()`

The top-level `express()` function creates an `Application` instance.

```javascript
import express from 'express';
const app = express();
```

### Top-Level Exports (Middleware & Constructors)

`express` exports several built-in middleware and core constructors:

*   **`express.json([options])`**: Parses incoming requests with JSON payloads. Based on `body-parser`.
*   **`express.raw([options])`**: Parses incoming requests with raw `Buffer` payloads. Based on `body-parser`.
*   **`express.static(root, [options])`**: Serves static assets from a `root` directory.
*   **`express.text([options])`**: Parses incoming requests with plain text payloads. Based on `body-parser`.
*   **`express.urlencoded([options])`**: Parses incoming requests with URL-encoded payloads. Based on `body-parser`.
*   **`express.Route`**: Constructor for creating a new Route instance. Used internally by `app.route()`.
*   **`express.Router([options])`**: Constructor for creating a new Router instance.
*   **`express.application`**: The prototype for `app` objects. Can be extended directly.
*   **`express.request`**: The prototype for `req` objects. Can be extended directly.
*   **`express.response`**: The prototype for `res` objects. Can be extended directly.

### `Application` (`app`)

The `app` object is the core of an Express application.

#### Configuration

*   **`app.set(setting, [value])`**: Assigns `value` to `setting`, or returns `setting`'s value if `value` is omitted.
    *   `setting`: `string`
    *   `value`: `any`
    *   Returns: `app` for chaining, or `setting`'s value.
*   **`app.get(setting)`**: Returns the value of `setting`.
    *   `setting`: `string`
    *   Returns: `any` value of the setting.
*   **`app.enable(setting)`**: Sets `setting` to `true`.
    *   `setting`: `string`
    *   Returns: `app` for chaining.
*   **`app.disable(setting)`**: Sets `setting` to `false`.
    *   `setting`: `string`
    *   Returns: `app` for chaining.
*   **`app.enabled(setting)`**: Checks if `setting` is `true`.
    *   `setting`: `string`
    *   Returns: `boolean`.
*   **`app.disabled(setting)`**: Checks if `setting` is `false`.
    *   `setting`: `string`
    *   Returns: `boolean`.
*   **`app.locals`**: An object containing local variables for the application, accessible in all templates rendered by the application.
    *   Type: `object`.
*   **`app.mountpath`**: The path pattern(s) on which a sub-application was mounted.
    *   Type: `string`.
*   **`app.path()`**: Returns the canonical URL path of the application, including mount paths of parent applications.
    *   Returns: `string`.

#### Routing & Middleware

*   **`app.use([path], middlewareFunction(s))`**: Mounts the specified middleware function(s) or `Application` instance(s) at the given `path`.
    *   `path`: `string` | `RegExp` | `Array<string | RegExp>` (optional, defaults to `/`).
    *   `middlewareFunction(s)`: `Function` | `Array<Function>` | `Application` | `Array<Application>`.
    *   Returns: `app` for chaining.
*   **`app.route(path)`**: Returns a new `Route` instance for the given `path`, allowing you to chain method handlers.
    *   `path`: `string`
    *   Returns: `express.Route` instance.
*   **`app.VERB(path, [callback...], handler)`**: Routes HTTP `VERB` requests to the specified `path` with callback functions. `VERB` can be `get`, `post`, `put`, `delete`, `patch`, `options`, `head`, `all`, etc.
    *   `path`: `string` | `RegExp` | `Array<string | RegExp>`.
    *   `callback...`: `Function` (middleware or handler).
    *   `handler`: `Function` (final handler).
    *   Returns: `app` for chaining.
*   **`app.param(name, handler)`**: Maps parameter placeholder logic for routes.
    *   `name`: `string` | `Array<string>` (parameter name, e.g., `'id'`).
    *   `handler`: `function(req, res, next, value, name)` (callback function for the parameter).
    *   Returns: `app` for chaining.

#### View Engine

*   **`app.engine(ext, callback)`**: Registers a template engine `callback` for the given file extension `ext`.
    *   `ext`: `string` (e.g., `'.html'`, `'ejs'`).
    *   `callback`: `function(path, options, callback)` (engine rendering function).
    *   Returns: `app` for chaining.
*   **`app.render(name, [options], callback)`**: Renders a view file and sends the rendered HTML string to the callback.
    *   `name`: `string` (view name or absolute path).
    *   `options`: `object` (optional, locals for the template).
    *   `callback`: `function(err, html)` (receives error and rendered HTML).

#### Server Management

*   **`app.listen([port[, host[, backlog]]][, callback])`**: Starts an HTTP server and listens for connections.
    *   Returns: `http.Server` instance.

### `Request` (`req`)

The `req` object represents the HTTP request.

*   **`req.get(field)` / `req.header(field)`**: Returns the specified HTTP header field value. `Referrer` and `Referer` are interchangeable.
    *   `field`: `string` (header name).
    *   Returns: `string` or `undefined`.
*   **`req.accepts(types...)`**: Checks if the specified content `type(s)` are acceptable.
    *   `types`: `string` | `Array<string>` (MIME type or extension).
    *   Returns: Best matching type `string`, `Array` of acceptable types, or `false`.
*   **`req.acceptsCharsets(charsets...)`**: Checks if the specified character `charset(s)` are acceptable.
    *   `charsets`: `string` | `Array<string>`.
    *   Returns: Best matching charset `string`, `Array` of acceptable charsets, or `false`.
*   **`req.acceptsEncodings(encodings...)`**: Checks if the specified `encoding(s)` are acceptable.
    *   `encodings`: `string` | `Array<string>`.
    *   Returns: Best matching encoding `string`, `Array` of acceptable encodings, or `false`.
*   **`req.acceptsLanguages(languages...)`**: Checks if the specified `language(s)` are acceptable.
    *   `languages`: `string` | `Array<string>`.
    *   Returns: Best matching language `string`, `Array` of acceptable languages, or `false`.
*   **`req.range(size, [options])`**: Parses the `Range` header field, capping to `size`.
    *   `size`: `number` (resource length).
    *   `options`: `object` (e.g., `combine: boolean`).
    *   Returns: `number` (`-1` unsatisfiable, `-2` invalid) or `Array<object>` (range objects with `start`, `end`, `type`).
*   **`req.query`**: An object containing a property for each query string parameter.
    *   Type: `object`.
*   **`req.is(type)`**: Checks if the incoming request's `Content-Type` header field matches the given `type`.
    *   `type`: `string` | `Array<string>` (MIME type or extension).
    *   Returns: Matching MIME type `string`, `false`, or `null` (no content type).
*   **`req.protocol`**: The request protocol string, "http" or "https". Respects `X-Forwarded-Proto` if `trust proxy` is enabled.
    *   Type: `string`.
*   **`req.secure`**: Boolean indicating if the connection is HTTPS.
    *   Type: `boolean`.
*   **`req.ip`**: The remote IP address of the request. Respects `X-Forwarded-For` if `trust proxy` is enabled.
    *   Type: `string`.
*   **`req.ips`**: An array of IP addresses from `X-Forwarded-For` header, if `trust proxy` is enabled.
    *   Type: `Array<string>`.
*   **`req.subdomains`**: An array of subdomains in the domain name.
    *   Type: `Array<string>`.
*   **`req.path`**: The URL pathname of the request.
    *   Type: `string`.
*   **`req.host`**: The full host (hostname:port) of the request. Respects `X-Forwarded-Host` if `trust proxy` is enabled.
    *   Type: `string`.
*   **`req.hostname`**: The hostname of the request.
    *   Type: `string`.
*   **`req.fresh`**: Boolean indicating if the request is "fresh" (cached).
    *   Type: `boolean`.
*   **`req.stale`**: Boolean indicating if the request is "stale" (not cached).
    *   Type: `boolean`.
*   **`req.xhr`**: Boolean indicating if the request was an XMLHttpRequest.
    *   Type: `boolean`.
*   **`req.params`**: An object containing properties mapped to the named route parameters.
    *   Type: `object`.
*   **`req.app`**: A reference to the `Application` instance that invoked the middleware.
    *   Type: `Application`.

### `Response` (`res`)

The `res` object represents the HTTP response.

*   **`res.status(code)`**: Sets the HTTP status `code` for the response. Must be an integer between 100 and 999.
    *   `code`: `number` (HTTP status code).
    *   Returns: `res` for chaining.
*   **`res.links(obj)`**: Sets the `Link` HTTP header field with the given `obj`.
    *   `obj`: `object` (key-value pairs where value is URL string or array of URL strings).
    *   Returns: `res` for chaining.
*   **`res.send(body)`**: Sends the HTTP response. `body` can be a `string`, `number`, `boolean`, `object` (JSON), or `Buffer` (`Uint8Array`).
    *   `body`: `string` | `number` | `boolean` | `object` | `Buffer` | `Uint8Array`.
    *   Returns: `res` for chaining.
*   **`res.json(obj)`**: Sends a JSON response.
    *   `obj`: `any` (object to be serialized to JSON).
    *   Returns: `res` for chaining.
*   **`res.jsonp(obj)`**: Sends a JSON response with JSONP support.
    *   `obj`: `any` (object to be serialized to JSON).
    *   Returns: `res` for chaining.
*   **`res.sendStatus(statusCode)`**: Sets the response status code and sends the corresponding status message as the body.
    *   `statusCode`: `number`.
    *   Returns: `res` for chaining.
*   **`res.sendFile(path, [options], [callback])`**: Transfers the file at `path` as an attachment.
    *   `path`: `string` (absolute path to file).
    *   `options`: `object` (optional, e.g., `root`, `maxAge`, `headers`).
    *   `callback`: `function(err)` (optional, invoked on transfer completion or error).
    *   Returns: `res` for chaining.
*   **`res.download(path, [filename], [options], [callback])`**: Transfers the file at `path` as an attachment.
    *   `path`: `string` (path to file).
    *   `filename`: `string` (optional, alternate filename).
    *   `options`: `object` (optional, passed to `res.sendFile()`).
    *   `callback`: `function(err)` (optional, invoked on transfer completion or error).
    *   Returns: `res` for chaining.
*   **`res.contentType(type)` / `res.type(type)`**: Sets the `Content-Type` HTTP header to the specified MIME `type`.
    *   `type`: `string` (MIME type or extension).
    *   Returns: `res` for chaining.
*   **`res.format(obj)`**: Responds to acceptable formats using an object of MIME-type callbacks.
    *   `obj`: `object` (callbacks keyed by MIME type or extension).
    *   Returns: `res` for chaining.
*   **`res.attachment([filename])`**: Sets the `Content-Disposition` header to `attachment`, optionally with a `filename`.
    *   `filename`: `string` (optional).
    *   Returns: `res` for chaining.
*   **`res.append(field, value)`**: Appends `value` to the `field` header. If the header is not already set, it creates the header with the given value.
    *   `field`: `string` (header name).
    *   `value`: `string` | `Array<string>`.
    *   Returns: `res` for chaining.
*   **`res.set(field, [value])` / `res.header(field, [value])`**: Sets the HTTP header `field` to `value`, or multiple fields if `field` is an object.
    *   `field`: `string` | `object`.
    *   `value`: `string` | `Array<string>` (optional).
    *   Returns: `res` for chaining.
*   **`res.get(field)`**: Returns the value of the specified response header `field`.
    *   `field`: `string`.
    *   Returns: `string`.
*   **`res.clearCookie(name, [options])`**: Clears the cookie `name`.
    *   `name`: `string`.
    *   `options`: `object` (optional, e.g., `path`).
    *   Returns: `res` for chaining.
*   **`res.cookie(name, value, [options])`**: Sets cookie `name` to `value`.
    *   `name`: `string`.
    *   `value`: `string` | `object`.
    *   `options`: `object` (optional, e.g., `maxAge`, `expires`, `signed`, `httpOnly`, `secure`, `partitioned`, `priority`).
    *   Returns: `res` for chaining.
*   **`res.location(url)`**: Sets the `Location` HTTP header to `url`.
    *   `url`: `string`.
    *   Returns: `res` for chaining.
*   **`res.redirect([status], url)`**: Redirects to `url` with optional `status` code (defaults to 302).
    *   `status`: `number` (optional HTTP status code).
    *   `url`: `string`.
    *   Returns: `res` for chaining.
*   **`res.vary(field)`**: Adds the specified `field` to the `Vary` header.
    *   `field`: `string` | `Array<string>`.
    *   Returns: `res` for chaining.
*   **`res.render(name, [options], [callback])`**: Renders a view file.
    *   `name`: `string` (view file name).
    *   `options`: `object` (optional, locals for the template).
    *   `callback`: `function(err, html)` (optional, if provided, response is not sent automatically).

## 4. Getting Started & Usage

This section guides you through setting up an Express.js project and demonstrates common usage patterns with copy-pasteable code examples.

### Prerequisites

*   Node.js version 18 or higher.
*   `npm` (Node Package Manager).

### Installation

To add Express.js to your project, use `npm install`:

```bash
# Create a new project directory and navigate into it
mkdir my-express-app
cd my-express-app

# Initialize a new Node.js project (accept defaults or customize)
npm init -y

# Install Express.js
npm install express
```

### Quick Start: Hello World

Create a file named `app.js` (or `index.js`) and add the following code:

```javascript
// app.js
import express from 'express';

const app = express();
const port = 3000;

app.get('/', (req, res) => {
  res.send('Hello World!');
});

app.listen(port, () => {
  console.log(`Server is running on http://localhost:${port}`);
});
```

Run your application:

```bash
node app.js
```

Open your browser and navigate to `http://localhost:3000`. You should see "Hello World!".

### Basic Routing

Express allows you to define routes for different HTTP methods and URL paths.

```javascript
// app.js (continued)
// ...

app.post('/items', (req, res) => {
  res.status(201).send('Item created.');
});

app.put('/items/:id', (req, res) => {
  const itemId = req.params.id;
  res.send(`Updated item ${itemId}.`);
});

app.delete('/items/:id', (req, res) => {
  const itemId = req.params.id;
  res.send(`Deleted item ${itemId}.`);
});

// ...
```

### Middleware

Middleware functions are executed sequentially and can perform various tasks, such as parsing request bodies, serving static files, logging, or authentication.

#### Parsing Request Bodies

Use `express.json()` and `express.urlencoded()` to automatically parse JSON and URL-encoded request bodies.

```javascript
// app.js (continued)
// ...
import express from 'express';
// ...

const app = express();

// Enable JSON body parsing for all incoming requests
app.use(express.json());
// Enable URL-encoded body parsing
app.use(express.urlencoded({ extended: true }));

app.post('/data', (req, res) => {
  console.log('Received data:', req.body);
  res.json({ received: req.body });
});

// ...
```

To test:
```bash
curl -X POST -H "Content-Type: application/json" -d '{"message": "Hello"}' http://localhost:3000/data
# Expected output: {"received":{"message":"Hello"}}

curl -X POST -H "Content-Type: application/x-www-form-urlencoded" -d 'name=Alice&age=30' http://localhost:3000/data
# Expected output: {"received":{"name":"Alice","age":"30"}}
```

#### Serving Static Files

Use `express.static()` to serve images, CSS files, JavaScript files, etc., from a designated directory.

```javascript
// app.js (continued)
// ...
import express from 'express';
import path from 'node:path'; // Import path module
// ...

const app = express();

// Serve static files from the 'public' directory
app.use(express.static(path.join(__dirname, 'public')));
// You can also mount static files under a virtual path prefix
app.use('/assets', express.static(path.join(__dirname, 'public')));

// ...
```

Create a `public` directory in your project root with some files, e.g., `public/index.html` and `public/css/style.css`.
Access them at `http://localhost:3000/index.html` or `http://localhost:3000/assets/css/style.css`.

#### Custom Middleware

You can write your own middleware functions for custom logic.

```javascript
// app.js (continued)
// ...

// Simple logger middleware
app.use((req, res, next) => {
  console.log(`${new Date().toISOString()} - ${req.method} ${req.url}`);
  next(); // Pass control to the next middleware/route handler
});

// Authentication middleware
function authenticate(req, res, next) {
  const apiKey = req.query['api-key'];
  if (apiKey === 'your-secret-api-key') {
    next(); // Authenticated, proceed
  } else {
    res.status(401).send('Unauthorized: Invalid API Key');
  }
}

app.get('/protected', authenticate, (req, res) => {
  res.send('Access granted to protected resource!');
});

// ...
```

### Templating

Express supports over 14 template engines. Here's an example using EJS (`npm install ejs`):

```javascript
// app.js (continued)
// ...
import express from 'express';
import path from 'node:path';
// ...

const app = express();

// Set the views directory
app.set('views', path.join(__dirname, 'views'));
// Set the view engine to ejs (or .html for ejs files with .html extension)
app.set('view engine', 'ejs');

// Example route rendering a view
app.get('/profile/:name', (req, res) => {
  res.render('profile', { title: 'User Profile', username: req.params.name });
});

// ...
```

Create a `views` directory with `views/profile.ejs`:
```ejs
<!-- views/profile.ejs -->
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title><%= title %></title>
</head>
<body>
  <h1>Welcome, <%= username %>!</h1>
  <p>This is your profile page.</p>
</body>
</html>
```

### Error Handling

Error-handling middleware functions have four arguments `(err, req, res, next)`.

```javascript
// app.js (continued)
// ...

// A route that intentionally throws an error
app.get('/error-route', (req, res, next) => {
  // Simulate an error
  const err = new Error('Something went wrong!');
  err.status = 500;
  next(err); // Pass the error to the error-handling middleware
});

// Generic error handler (must be the last middleware)
app.use((err, req, res, next) => {
  console.error(err.stack); // Log the error stack for debugging
  res.status(err.status || 500);
  res.send(`Error: ${err.message}`);
});

// Handle 404 Not Found (if no other route matched)
app.use((req, res, next) => {
  res.status(404).send("Sorry, can't find that!");
});

// ...
```

### Router for Modular Code

For larger applications, it's beneficial to organize routes into separate modules using `express.Router()`.

```javascript
// routes/users.js
import express from 'express';
const router = express.Router();

// Middleware specific to this router
router.use((req, res, next) => {
  console.log('Time:', Date.now());
  next();
});

// Define routes
router.get('/', (req, res) => {
  res.send('List of users');
});

router.get('/:id', (req, res) => {
  res.send(`User ${req.params.id}`);
});

export default router;

// app.js
// ...
import express from 'express';
import usersRouter from './routes/users.js'; // Assuming you use ES Modules

const app = express();

app.use('/users', usersRouter); // Mount the router at /users

// ...
```

### Sending Different Response Types

Express provides flexible methods for sending various data formats.

```javascript
// app.js (continued)
// ...

// Sending JSON
app.get('/api/data', (req, res) => {
  res.json({ message: 'This is JSON data', timestamp: new Date() });
});

// Sending JSONP (if 'callback' query parameter is present)
app.get('/api/jsonp', (req, res) => {
  res.jsonp({ data: 'JSONP response' });
});

// Sending a file as a download
app.get('/download/report', (req, res) => {
  const filePath = path.join(__dirname, 'reports', 'sales-report.pdf');
  res.download(filePath, 'sales-report-2023.pdf', (err) => {
    if (err) {
      console.error('File download failed:', err);
      res.status(500).send('Could not download file.');
    }
  });
});

// Redirecting
app.get('/old-path', (req, res) => {
  res.redirect('/new-path'); // 302 Found by default
});

app.get('/permanent-redirect', (req, res) => {
  res.redirect(301, '/new-permanent-path'); // 301 Moved Permanently
});

// Content Negotiation with res.format()
app.get('/negotiate', (req, res) => {
  res.format({
    'text/plain': () => {
      res.send('Hello from plain text!');
    },
    'text/html': () => {
      res.send('<h1>Hello from HTML!</h1>');
    },
    'application/json': () => {
      res.json({ message: 'Hello from JSON!' });
    },
    default: () => {
      // Respond with 406 Not Acceptable if no matching type
      res.status(406).send('Not Acceptable');
    }
  });
});
```

## 5. Security & Performance

Express.js is designed with security and performance in mind, offering features and practices that enable developers to build robust and efficient web applications.

### Security Posture

1.  **Automated Security Scanning**: The project integrates **GitHub CodeQL** and **OpenSSF Scorecard** into its CI/CD workflows (`.github/workflows/codeql.yml`, `.github/workflows/scorecard.yml`).
    *   **CodeQL**: Analyzes JavaScript code for potential vulnerabilities, helping maintain a high standard of code security.
    *   **OpenSSF Scorecard**: Evaluates supply-chain security practices, providing insights into the project's security health and adherence to best practices.

2.  **Dependency Management**: **Dependabot** (`.github/dependabot.yml`) is configured to automatically check for and update vulnerable dependencies monthly for both npm packages and GitHub Actions, ensuring the project uses up-to-date and secure versions of its external libraries. Major version updates are explicitly ignored to prevent breaking changes unless manually reviewed.

3.  **Vulnerability Response**: The project demonstrates a commitment to security by promptly addressing reported vulnerabilities.
    *   Version `5.0.1` included an update to the `cookie` dependency to address **CVE-2024-47764**.
    *   Version `5.2.1` reverted an erroneous security fix for **CVE-2024-51999** (which was subsequently rejected), demonstrating a careful approach to changes that might impact stability while addressing security concerns.

4.  **Input Validation and Sanitization**:
    *   **`res.status(code)`**: Strict validation ensures that status codes are integers between 100 and 999, preventing malformed responses.
    *   **`res.location(url)` / `res.redirect(url)`**: These methods incorporate URL encoding (`encodeurl` dependency) to prevent common redirect allow-list bypasses and XSS vulnerabilities, ensuring that URLs are properly sanitized before being used in Location headers.
    *   **JSONP (`res.jsonp`)**: Includes specific security mitigations against "Rosetta Flash JSONP abuse" by prepending `/**/` and restricting callback characters, along with setting `X-Content-Type-Options: nosniff` to prevent content sniffing attacks.
    *   **Query Parser**: The `qs` module used for extended query parsing (`qs@^6.14.2`) is configured with `allowPrototypes: true`, which is a deliberate choice for Express's flexibility. While `Object.prototype` pollution is a known attack vector, Express's internal handling and regular updates to `qs` help mitigate risks. Developers using `req.query` directly should be aware of this and validate input.
    *   **`app.set()`**: The application's `set` method ignores `Object.prototype` values when setting properties, preventing prototype pollution through configuration.

5.  **Proxy Trust Configuration**: The `app.set('trust proxy', value)` setting (`proxy-addr` dependency) is crucial for applications running behind reverse proxies. It allows Express to correctly interpret client IP addresses (`req.ip`, `req.ips`), protocols (`req.protocol`, `req.secure`), and hostnames (`req.hostname`) from `X-Forwarded-*` headers, preventing IP spoofing and ensuring accurate logging and security checks.

6.  **Static File Serving**: `express.static` and underlying `send` module set a strict `Content-Security-Policy: default-src 'none'` header on redirect and error responses, preventing execution of inline scripts and unauthorized content loading.

### Performance Optimizations

1.  **Lean Core and Middleware**: Express itself is a minimalist framework, focusing on providing essential HTTP server utilities. This "unopinionated" approach allows developers to choose only the middleware they need, minimizing overhead.
    *   Many functionalities are offloaded to highly optimized, external modules (e.g., `body-parser`, `send`, `router`, `etag`, `fresh`, `mime-types`).

2.  **Efficient Routing**: The underlying `router` module is designed for performance, including:
    *   Fast matching paths for common patterns like `*`.
    *   Optimizations for handling large middleware and route stacks, mitigating stack overflow issues in synchronous routes.
    *   Promise support in routes and middleware allows for cleaner asynchronous code without blocking the event loop.

3.  **Caching Mechanisms**:
    *   **View Cache**: `app.enable('view cache')` in production environments (`app.defaultConfiguration`) caches compiled templates, reducing file system I/O and compilation time on subsequent renders.
    *   **ETag Generation**: Express automatically generates `ETag` headers for responses (`res.send()`, `res.sendFile()`) when the `etag` setting is enabled (defaulting to `weak`). This enables efficient client-side caching and conditional requests (`If-None-Match`, `If-Modified-Since`), leading to 304 Not Modified responses that avoid transferring redundant data. The `etag` library uses SHA1 for hashing, which is efficient and FIPS 140-2 compatible.
    *   **`maxAge` for Static Files**: `express.static` allows setting `maxAge` for `Cache-Control` headers, instructing browsers to cache assets for a specified duration, reducing subsequent requests to the server.
    *   **`fresh` module**: Used internally to determine if a response is "fresh" (`req.fresh`), allowing for 304 Not Modified responses when appropriate.

4.  **Optimized I/O**:
    *   **`res.send()`**: Intelligently handles various body types (string, Buffer, object), optimizing content-type setting and length calculation. For small string responses without explicit `Content-Type`, it avoids duplicate header processing. `Uint8Array` support enhances flexibility for binary data.
    *   **`res.sendFile()`**: Leverages the highly optimized `send` module for efficient file transfer, including support for range requests, `Last-Modified` headers, and streaming.
    *   **Buffer Handling**: Efficient use of `Buffer` for string conversion and length calculations, especially when ETag generation is not needed or for small chunks.

5.  **Minimal Overhead**:
    *   Express avoids unnecessary operations, for example, `res.send()` strips irrelevant headers (e.g., `Content-Type`, `Content-Length`, `Transfer-Encoding`) for 204/304 responses, and sets `Content-Length: 0` for 205 responses.
    *   Default behavior for `req.query` parsing is 'simple' (`querystring` module) unless 'extended' (`qs` module) is explicitly configured, providing a performance trade-off for query parsing complexity.

## 6. Contribution Guide

We welcome contributions to Express.js! This guide provides the necessary steps to set up your development environment, run tests, and adhere to our coding standards.

### Prerequisites

Ensure you have the following installed:
*   **Node.js**: Version 18 or higher. (Older Node.js versions, like 16 & 17, are tested in CI for legacy support, but development should happen on `^18`.)
*   **Git**: For cloning the repository.
*   **npm**: Node Package Manager, which comes with Node.js.

### Setting Up Your Development Environment

1.  **Clone the repository**:
    ```bash
    git clone https://github.com/expressjs/express.git
    cd express
    ```
2.  **Install dependencies**:
    ```bash
    npm install
    ```
    This command installs both production and development dependencies. The `ci.yml` workflow uses `npm install --ignore-scripts --include=dev` for linting, but `npm install` is sufficient for full local development.

### Linting

We enforce strict code style and quality using ESLint.
*   **Configuration**: ESLint rules are defined in `.eslintrc.yml`. Key rules include:
    *   2-space indentation.
    *   `eqeqeq` (strict equality `===`).
    *   No trailing spaces.
    *   No unused variables.
    *   Restricted global `Buffer` usage, requiring `import { Buffer } from "node:buffer"` for clarity and consistency.
*   **Running the linter**:
    ```bash
    npm run lint
    ```
*   **Automatically fix linting issues**:
    ```bash
    npm run lint:fix
    ```
*   **CI Integration**: The `lint` job in `.github/workflows/ci.yml` runs ESLint on every push and pull request to ensure code quality before merging.

### Testing

The project uses Mocha for testing and NYC for code coverage.

*   **Running all tests**:
    ```bash
    npm test
    ```
    This command runs both unit tests and acceptance tests with the default spec reporter.
*   **Continuous Integration Tests (with coverage)**:
    ```bash
    npm run test-ci
    ```
    This script is used in our CI environment (`.github/workflows/ci.yml`) and generates `lcov.info` for coverage reporting to Coveralls.
*   **Local Coverage Report**:
    ```bash
    npm run test-cov
    ```
    This generates an HTML coverage report in the `./coverage` directory, which you can view in your browser.
*   **TAP Reporter**:
    ```bash
    npm run test-tap
    ```
    Runs tests with the Test Anything Protocol (TAP) reporter.

*   **CI Integration**:
    *   The `test` job in `.github/workflows/ci.yml` runs tests on `ubuntu-latest` and `windows-latest` across a matrix of Node.js versions (18, 19, 20, 21, 22, 23, 24, 25).
    *   The `legacy` workflow (`.github/workflows/legacy.yml`) specifically tests older Node.js versions (16, 17) to ensure broader compatibility.
    *   The `coverage` job aggregates coverage reports from all `test` jobs and uploads them to Coveralls.

### Contributing Workflow

1.  **Fork the repository**: Create your own fork of the Express.js repository on GitHub.
2.  **Create a new branch**: Branch off `master` for new features or bug fixes. Use descriptive branch names (e.g., `feature/add-new-middleware`, `fix/bug-in-router`).
3.  **Implement your changes**: Write your code, ensuring it adheres to the project's coding standards.
4.  **Write tests**: Add unit and/or acceptance tests for your changes to maintain high test coverage.
5.  **Run tests and linting locally**: Before committing, ensure all tests pass and linting issues are resolved.
    ```bash
    npm run lint && npm test
    ```
6.  **Commit your changes**: Write clear, concise commit messages.
7.  **Push to your fork**: Push your branch to your GitHub fork.
8.  **Open a Pull Request**: Submit a pull request to the `master` branch of the official Express.js repository. Provide a detailed description of your changes and reference any related issues.

### Code of Conduct

Please review the [Code of Conduct](https://github.com/expressjs/.github/blob/HEAD/CODE_OF_CONDUCT.md) to understand the standards of behavior expected within the Express.js community.

### Security Issues

If you discover a security vulnerability, please refer to the project's [Security Policies and Procedures](https://github.com/expressjs/express/security/policy) for responsible disclosure guidelines.
